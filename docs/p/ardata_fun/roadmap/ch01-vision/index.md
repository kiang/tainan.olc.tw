# 第一章：願景、動機與使用者人物誌

---

## 1.1 為什麼這個系統需要存在？

### 問題的根源：公開但不可用的資料

台灣監察院依據《政治獻金法》要求所有候選人及政黨公開申報政治獻金。這些申報資料理論上任何人都可以查詢，然而現實是：

1. **介面設計不友善**：官方查詢介面一次只能查單一候選人，無法跨候選人比較
2. **資料量龐大但無結構**：每屆選舉動輒數十萬筆紀錄，無法人工逐一比對
3. **缺乏關聯脈絡**：獻金資料本身是孤立的，必須對照政府採購網、都市計畫公告等外部資料才能顯現意義
4. **技術門檻**：會寫程式的公民不一定熟悉資料脈絡；熟悉脈絡的公民（記者、社運人士）不一定會寫程式

**這個系統要解決的，是讓「有時間調查的普通人」能夠輕鬆進入這份公開資料，並把他們發現的線索結構化地記錄下來。**

### 前金後謝：最常見的可疑模式

政治獻金問題的核心模式通常有兩種：

**模式 A：前金後謝（賄賂型）**
```
公司捐款 → 政治人物當選/當政 → 公司得標/獲批
     └── 時間差通常 3-18 個月 ──┘
```

**模式 B：後謝前金（感謝型）**
```
政治人物決策有利於公司 → 公司大額捐款 → 選舉期間集中出現
```

兩種模式的共同特徵：**捐款金額異常大**、**捐款時間點與特定政策決定高度相關**、**捐款者在相關領域有業務利益**。

## 1.2 核心價值主張

本系統的設計哲學：

> **把困難的資料處理留給機器，把有意義的判斷留給人。**

機器做的事：
- 下載並整理海量申報資料
- 提供快速搜尋與篩選
- 自動帶入上下文資訊到回報表單

人做的事：
- 判斷某筆捐款是否「在奇怪的時間點」出現
- 找到對應的標案或政策決定
- 提供可查證的公開連結

## 1.3 使用者人物誌（User Personas）

### Persona 1：阿明 — 關心地方政治的市民

**背景**
- 45歲，台南市民，做小生意
- 不懂程式，但會用 Facebook 和 LINE
- 曾看過新聞報導說某政治人物的金主得了大標案

**動機**
- 他的社區有一塊地要被重劃，他想知道背後是否有金主
- 想知道那個常來拜票的議員的金主是誰

**使用行為**
- 用手機開啟系統
- 輸入議員名字，看看誰捐了錢
- 看到某建設公司捐了 200 萬，點「回報關聯案件」
- 填入「某某重劃案，2019年核准」

**設計需求**
- 手機螢幕友善
- 不需要帳號即可使用
- 回報流程不超過 3 個步驟

---

### Persona 2：小玲 — 獨立新聞工作者

**背景**
- 32歲，自由撰稿人，為幾個調查媒體寫稿
- 會用試算表，稍微懂一點資料分析
- 平常用政府採購網追蹤大型標案

**動機**
- 正在調查某特定產業（如砂石、廢棄物處理）的廠商與政治人物的關係
- 需要快速交叉比對某公司是否同時捐款給多位政治人物

**使用行為**
- 輸入統一編號直接搜尋特定公司
- 篩選「100萬以上」，縮小範圍
- 把搜尋結果分享連結貼給編輯確認
- 對每一筆可疑紀錄填寫詳細的回報說明

**設計需求**
- 能用統一編號精確搜尋
- 可以產生帶有篩選條件的分享連結
- 回報說明欄位有足夠空間填寫詳細脈絡

**背景**

## 1.4 反使用者（Anti-personas）：我們不服務誰

設計系統時，同樣重要的是明確「不為誰設計」：

**政治公關人員**：試圖用系統找出「哪些捐款紀錄已被公眾注意」，以便提前部署反制
- 對策：系統回報資料不公開展示（只匯入 Google Sheets 供管理者審核）

**惡意檢舉者**：捏造或誇大「關聯案件」，攻擊政治對手
- 對策：回報表單要求填入可查證的公開連結；說明免責聲明清楚標示「初步合理懷疑，非定論」

---

## 1.5 成功的定義

### 定量指標（Phase 3 結束時）
- 累積有效回報 > 100 筆
- 每週活躍使用者 > 50 人
- 平均每筆回報附有至少一個可查證連結

### 定性指標
- 至少一篇調查媒體報導引用本系統的資料
- 公民社會討論中出現「去政治獻金搜查系統查一下」的自發行為

## 1.6 為什麼有趣比正確更重要

做公民科技的人常常犯一個錯誤：把「重要性」當作人們參與的理由。

「這件事很重要，所以大家應該要幫忙。」

這個邏輯在現實中行不通。政治獻金資料固然重要，但「重要」並不能讓人在下班後打開電腦，花半小時去查一個陌生公司的捐款紀錄。人的注意力是稀缺的，競爭對手是 Netflix、社群媒體、朋友聚會。

看看成功的群眾外包案例怎麼做的：

**Galaxy Zoo**（2007年）：天文學家把兩百萬張星系照片放上網，請大家幫忙分類。他們沒有說「這對科學很重要，請幫忙」。他們設計了一個像遊戲的介面——點擊、分類、下一張。加上即時回饋（「你剛才分類了一個螺旋星系！」），六個月內有 15 萬名志願者分類了一百萬張照片，做到專業天文學家要花三年才能完成的工作。

**Foldit**（2008年）：蛋白質折疊是一個超級困難的科學問題，電腦模擬花了很多年都沒突破。研究人員把它做成一個益智遊戲。玩家不需要懂生化，只需要「把這個形狀拼好看」。結果：一群遊戲玩家在三週內解決了一個困擾 AIDS 研究者十五年的蛋白質結構問題。

**g0v（零時政府）**：台灣的公民科技社群。他們的黑客松不是義工活動，而是節慶。大家帶著好奇心、炫技心態、交朋友的慾望來參加，順便解決了一些公共問題。「先問問題，再寫 code」的文化讓參與者感覺自己是在創造，而不是在服從。

這三個案例的共同點：**參與行為本身就是令人愉快的**。人們不是因為「這件事重要」才做，而是因為「這件事有趣、讓我感覺自己在做有意義的事、還能被別人看見」。

本系統的挑戰是：把「查政治獻金資料 + 填回報表單」這兩件聽起來無聊的事，包裝成讓人欲罷不能的體驗。

偵探敘事框架是我們的答案。每個使用者都是公民偵探。每筆捐款都是線索。每次回報都是破案。成就徽章、職階晉升、分享功能——這些不是裝飾，而是讓參與行為本身產生意義感和身份認同的機制。

當一個人可以在朋友面前說「我是首席偵探，已經破了三十個案子」的時候，他已經不是在「幫忙做一件重要的事」，而是在「做一件屬於自己的事」。這才是讓群眾外包真正運作的力量。
